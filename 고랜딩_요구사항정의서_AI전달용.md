# 고랜딩 요구사항 정의서 (AI 전달용)

## 1. 문서 목적

이 문서는 다른 AI 또는 다른 개발자가 고랜딩의 요구사항을 빠르게 이해하고, 같은 방향으로 추가 작업을 이어가기 위한 기준 문서다.

이 문서의 목표는 다음과 같다.

- 제품이 무엇인지 한 문장으로 정의한다.
- 반드시 구현해야 하는 기능과 제외 범위를 명확히 구분한다.
- 분석 규칙, 입력 규칙, 관리자 규칙을 오해 없이 고정한다.
- 이후 설계/구현/테스트 작업의 기준점으로 사용한다.

## 2. 제품 한 줄 정의

고랜딩은 사용자가 이미 만든 랜딩 자산을 등록하고 발행한 뒤, 방문자의 클릭, 스크롤, 체류를 수집하여 히트맵, 스크롤맵, 체류맵으로 분석하는 SaaS다.

## 3. 현재 단계에서의 제품 범위

현재 단계에서 포함되는 범위:

- 랜딩 등록
- 랜딩 발행
- 클릭 히트맵
- 스크롤맵
- 체류맵
- 관리자/분석 페이지
- DB 수집형 랜딩의 제출 데이터 저장
- 제출 데이터 CSV 다운로드
- 승인 이메일 기반 로그인
- 관리자 계정/승인 계정 관리

현재 단계에서 제외되는 범위:

- 랜딩 자동 생성 AI
- 광고 매체 연동 자동화
- 세션 리플레이
- 고급 퍼널 분석
- A/B 테스트
- 고급 실험 플랫폼 기능

## 4. 핵심 사용자

주 사용자:

- 랜딩을 등록하고 운영하는 운영자/마케터
- 메타광고 클래스 수강생 또는 승인된 creator 사용자
- 관리자 계정을 가진 운영자

방문자:

- 공개 랜딩 URL에 접속하는 일반 사용자

## 5. 핵심 용어 정의

- `creator`: 승인 이메일로 로그인하여 랜딩을 만들고 분석을 보는 사용자
- `admin`: 승인 계정 관리와 운영 제어가 가능한 관리자 사용자
- `landing`: 공개 발행 가능한 랜딩 단위
- `public slug`: 공개 랜딩 URL 식별자
- `session`: 방문자 1회의 접속 단위
- `click heatmap`: 클릭 좌표 누적 시각화
- `scroll map`: 페이지 깊이 도달률 시각화
- `dwell map`: 20등분 구간별 체류 비율 누적 시각화

## 6. 지원해야 하는 랜딩 유형

고랜딩은 정확히 아래 3가지 랜딩 유형을 지원해야 한다.

### 6.1 버튼형 랜딩

설명:

- 랜딩 이미지를 등록하고 CTA 버튼을 별도 블록으로 배치하는 형태

필수 요구사항:

- 이미지 1장 이상 등록 가능
- 버튼 1개 이상 등록 가능
- 버튼 다중 배치 가능
- 각 버튼은 다음 속성을 가져야 함
  - 라벨
  - 이동 URL
  - width ratio
  - 정렬/배치 제어가 가능한 구조
- 버튼 UI는 랜딩 이미지의 색상/분위기와 최대한 이질적이지 않아야 함

### 6.2 DB 수집형 랜딩

설명:

- 랜딩 이미지 아래 또는 주변에 폼을 붙여서 고객 정보를 수집하는 형태

필수 요구사항:

- 이미지 1장 이상 등록 가능
- 폼 필드를 사용자가 조합 가능
- 지원 필드:
  - 이름
  - 이메일
  - 전화번호
  - 주소
  - 메모1
  - 메모2
  - 메모3
- 모든 필드 라벨은 수정 가능해야 함
- 모든 필드는 필수/선택 설정 가능해야 함
- 메모형 필드는 질문/설명 용도의 사용자 정의 입력란으로 사용 가능해야 함
- 폼 UI는 랜딩 이미지의 색상/분위기와 최대한 이질적이지 않아야 함

### 6.3 HTML 소스형 랜딩

설명:

- 외부에서 직접 작성한 HTML/CSS/JS 소스를 등록하는 형태

필수 요구사항:

- 전체 HTML 소스 입력 가능
- 외부 소스라서 내부 스타일과 완전한 일치는 강제하지 않음
- 그래도 분석 기능은 동일하게 적용해야 함

## 7. 랜딩 이미지 규칙

- 랜딩 이미지는 여러 장 업로드 가능해야 함
- 여러 장 이미지는 순서대로 이어붙여 1개의 긴 랜딩처럼 렌더링해야 함
- 우선순위는 모바일 가독성이다
- 하나의 반응형 기준만 선택해야 한다면 모바일 기준으로 설계한다
- 향후 PC/모바일 분리 버전이 생기더라도 모바일을 주 기준으로 본다

## 8. 프론트 스타일 조절 범위

버튼형과 DB 수집형에서 고랜딩이 직접 렌더링하는 UI는 최소한 아래 항목을 조절할 수 있어야 한다.

- primary color
- text color
- surface/background color
- border radius

목적:

- 완전한 디자인 편집기가 아니라, 업로드된 랜딩 이미지와 명백히 어색하지 않은 수준의 톤 맞춤

## 9. 인증 및 접근 규칙

현재 인증 모델:

- 승인 이메일 기반 로그인
- 서버 세션 기반 인증
- 승인된 계정만 creator 기능 사용 가능
- 관리자 이메일은 별도 env로 지정

필수 규칙:

- 승인 이메일이 아니면 로그인 불가
- 관리자 이메일은 운영 제어 가능
- production DB가 비어 있어도 env에 등록된 관리자 이메일은 자동 승인 계정으로 부트스트랩되어야 함

## 10. 관리자 페이지 요구사항

관리자/creator 화면에서 최소한 아래를 제공해야 한다.

### 10.1 랜딩 목록

- 제목
- 생성일
- 랜딩 유형
- 공개 URL
- 방문 수

### 10.2 승인 계정 관리

- 승인 계정 목록 보기
- 승인 계정 추가
- 승인 계정 수정
- 상태 변경
- CSV 일괄 등록
- CSV 내보내기

### 10.3 세션 관리

- 활성 creator 세션 목록 보기
- 강제 로그아웃

### 10.4 운영 상태

- 관리자 대시보드 요약 카드
- 배포 준비 상태 체크 패널

## 11. 분석 페이지 요구사항

분석 페이지는 최소한 아래를 보여야 한다.

- 클릭 히트맵
- 스크롤맵
- 체류맵
- 총 방문 수
- 총 클릭 수
- CTA 클릭 수
- 폼 제출 수

## 12. 클릭 히트맵 규칙

- 랜딩 전체 클릭을 추적해야 한다
- CTA 버튼 클릭은 별도로 식별 가능해야 한다
- DB 수집형은 폼 상호작용도 클릭 분석에 포함해야 한다
- HTML 소스형도 동일한 클릭 분석을 적용해야 한다

## 13. 스크롤맵 규칙

- 방문자 세션 기준으로 스크롤 깊이를 누적해야 한다
- 사용자가 페이지의 어느 정도까지 도달했는지 시각화해야 한다
- 강한 구간, 약한 구간, 이탈이 많은 구간을 파악할 수 있어야 한다

## 14. 체류맵 규칙

### 14.1 분할 규칙

- 모든 랜딩은 세로 기준 20등분한다
- 체류 분석은 항상 이 20개 구간을 기준으로 계산한다

### 14.2 방문자별 정규화 규칙

각 세션에 대해:

- 해당 세션의 유효 체류시간 총합을 100%로 본다
- 각 섹션 체류시간을 해당 세션의 비율로 환산한다
- 환산된 비율을 전체 방문자 기준으로 누적한다

예시 해석:

- 방문자 A 유효 체류시간 20초
- 3번 섹션 10초 체류 -> 50%
- 5번 섹션 5초 체류 -> 25%

- 방문자 B 유효 체류시간 30초
- 8번 섹션 15초 체류 -> 50%
- 19번 섹션 10초 체류 -> 33.3%

누적 목적:

- 가장 많이 본 구간
- 상대적으로 약한 구간
- 이탈이 큰 구간
- 섹션별 관심도 비교

### 14.3 세션 기준

- 체류맵 누적은 세션 단위다
- 같은 사용자가 다시 접속해 새 세션이 열리면 별도 방문으로 본다

## 15. 무반응 필터링 규칙

체류 계산에서 아래 규칙은 필수다.

### 15.1 30초 무반응 구간 제외

- 의미 있는 활동 없이 30초 방치된 시간은 유효 체류시간에 포함하지 않는다

### 15.2 60초 이상 무반응 세션 제외

- 60초 이상 의미 있는 활동이 없으면 해당 세션은 체류맵 누적에서 제외한다

현재 구현 해석:

- 의미 있는 활동 = 스크롤, 클릭, CTA 클릭, 폼 상호작용

## 16. DB 수집 데이터 규칙

- DB 수집형 랜딩은 제출 데이터를 저장해야 한다
- 관리자에서 제출 데이터 조회 가능해야 한다
- CSV 다운로드 가능해야 한다

CSV 최소 포함 항목:

- 랜딩 식별자
- 제출 시각
- 필드 라벨
- 제출 값

## 17. 공통 분석 적용 범위

아래 3개 타입 모두 동일한 분석 범위를 가져야 한다.

- 버튼형: 적용
- DB 수집형: 적용
- HTML 소스형: 적용

어떤 랜딩 유형도 분석에서 제외되지 않는다.

## 18. 비기능 요구사항

- SaaS 형태로 동작해야 한다
- 서버 세션 기반 보호가 되어야 한다
- 로컬 개발은 SQLite로 가능해야 한다
- 운영 배포는 PostgreSQL을 기준으로 해야 한다
- 운영 배포는 S3 호환 스토리지를 기준으로 해야 한다
- 관리자 화면에서 운영 readiness를 확인할 수 있어야 한다

## 19. 수용 기준

아래가 만족되면 현재 단계 요구사항을 충족한 것으로 본다.

- 사용자는 3가지 랜딩 유형 중 하나로 랜딩을 등록할 수 있다
- 다중 이미지 랜딩이 1개의 긴 랜딩처럼 렌더링된다
- 버튼형은 다중 버튼과 비율 조절을 지원한다
- DB 수집형은 필드 라벨 수정과 필수/선택 설정을 지원한다
- HTML 소스형도 발행 및 분석이 가능하다
- 모든 랜딩 유형에서 클릭/스크롤/체류 분석이 동작한다
- 체류맵은 20등분 기준으로 계산된다
- 체류맵은 방문자별 정규화 후 누적된다
- 30초 무반응 구간은 제외된다
- 60초 이상 무반응 세션은 제외된다
- 관리자에서 승인 계정/세션을 관리할 수 있다
- 승인 계정 CSV import/export가 가능하다
- 관리자 화면에서 운영 요약과 readiness를 볼 수 있다

## 20. AI가 추가 작업 시 지켜야 할 원칙

- 기존 3가지 랜딩 유형 정의를 바꾸지 말 것
- 체류맵의 20등분, 30초/60초 규칙을 변경하지 말 것
- 분석 기능은 모든 랜딩 타입에 공통 적용된다는 전제를 유지할 것
- 운영 배포는 PostgreSQL + S3 기준이라는 방향을 유지할 것
- 관리자 이메일 부트스트랩 로직을 제거하지 말 것
- 요구사항을 바꿔야 한다면 반드시 문서부터 수정한 뒤 구현을 바꿀 것
